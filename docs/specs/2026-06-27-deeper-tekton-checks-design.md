# Deeper Tekton Checks: 28 to 44

**Date**: 2026-06-27
**Status**: Draft
**Author**: Ugo Giordano

## Problem

tekton-guard has 28 checks but only covers 5 Tekton CRD kinds (PipelineRun, Pipeline, Task, TaskRun, StepAction). The full Tekton CRD surface includes TriggerTemplate, TriggerBinding, EventListener, Repository (PaC), and VerificationPolicy. These represent zero-competition attack surfaces that no existing tool scans.

Additionally, existing checks focus on individual resource properties (is this ref pinned?) but not on pipeline execution flow (can a security task be skipped? can results be poisoned?).

## Solution

Add 16 new checks across 4 categories, taking tekton-guard from 28 to 44 checks. Extend the parser to handle 4 new CRD kinds.

## Parser Extensions

Add to `TEKTON_KINDS`:
```python
TEKTON_KINDS = {
    "PipelineRun", "Pipeline", "Task", "TaskRun", "StepAction",
    "TriggerTemplate", "TriggerBinding", "EventListener",
    "Repository",
}
```

New fields on `TektonResource` for trigger CRDs:
- `trigger_template_params`: list of param definitions from TriggerTemplate
- `trigger_resource_templates`: list of resourcetemplates (PipelineRuns generated by triggers)
- `trigger_bindings`: list of TriggerBinding param extractions
- `event_listener_triggers`: list of triggers with binding/template pairs
- `event_listener_interceptors`: list of interceptor configs
- `repository_spec`: raw PaC Repository spec (allowed events, branches, etc.)

For VerificationPolicy, scan as raw YAML without special fields (check regex patterns directly).

## New Checks

### Trigger Deep (TKN-TRIG-004..007)

**TKN-TRIG-004: TriggerTemplate param injection**
- Severity: HIGH
- CWE: CWE-94
- Detect: TriggerTemplate has `resourcetemplates` containing a PipelineRun where params reference `$(tt.params.*)` that originate from TriggerBinding fields mapped to webhook body (e.g., `$(body.pull_request.title)`)
- Risk: webhook body content flows through TriggerBinding -> TriggerTemplate -> PipelineRun params -> Task scripts, enabling code injection
- Parser: extract TriggerTemplate resourcetemplates, find `$(tt.params.*)` interpolations
- Remediation: validate or sanitize trigger params before passing to PipelineRun

**TKN-TRIG-005: EventListener without interceptor**
- Severity: MEDIUM
- CWE: CWE-284
- Detect: EventListener with triggers that have no `interceptors` configured
- Risk: raw webhook payload reaches TriggerBinding without validation (no GitHub signature verification, no CEL filtering)
- Parser: check `spec.triggers[].interceptors` exists and is non-empty
- Remediation: add a webhook interceptor (GitHub, GitLab, CEL, or Bitbucket) to validate payload authenticity

**TKN-TRIG-006: PaC Repository allows all branches**
- Severity: HIGH
- CWE: CWE-284
- Detect: PaC `Repository` CRD with `spec.git_provider.secret` but no branch restrictions, or branch pattern set to `*` or empty
- Risk: any branch push triggers pipeline execution, including attacker-created branches
- Parser: check Repository spec for branch configuration
- Remediation: restrict to specific branches (main, release-*)

**TKN-TRIG-007: EventListener SA with excessive permissions**
- Severity: MEDIUM
- CWE: CWE-269
- Detect: EventListener references a ServiceAccount or uses the default SA (which may have broad permissions to create PipelineRuns cluster-wide)
- Parser: extract `spec.serviceAccountName` from EventListener, flag if default or missing
- Remediation: use a dedicated SA with minimal permissions (create PipelineRuns in specific namespace only)

### Supply Chain Deep (TKN-CHAIN-003..007)

**TKN-CHAIN-003: Results missing type hint for Chains**
- Severity: HIGH
- CWE: CWE-345
- Detect: Task that produces `IMAGE_URL` or `IMAGE_DIGEST` results but without `type: string` annotation. Chains uses type-hinting to identify which results to sign.
- Parser: check `spec.results[].name` for IMAGE_URL/IMAGE_DIGEST and verify `type` field
- Remediation: add `type: string` to result definitions consumed by Chains

**TKN-CHAIN-004: Build pipeline without finally signing block**
- Severity: MEDIUM
- CWE: CWE-345
- Detect: Pipeline labeled as build type that has no task in `finally` block with a security-relevant name (sign, attest, verify, enterprise-contract)
- Risk: signing/attestation tasks in the main task list can be skipped if an upstream task fails
- Parser: check `spec.finally[]` for security-named tasks
- Remediation: move signing and attestation tasks to the `finally` block

**TKN-CHAIN-005: VerificationPolicy with unanchored regex**
- Severity: HIGH
- CWE: CWE-185
- Detect: VerificationPolicy resource patterns that use regex without `^` and `$` anchors. CVE-2026-25542 showed that `foo` matches `foobar`, allowing policy bypass.
- Parser: scan VerificationPolicy YAML for `resourcePattern` fields, check regex anchoring
- Remediation: anchor all regex patterns with `^` and `$`

**TKN-CHAIN-006: Chains-consumed result from untrusted task**
- Severity: HIGH
- CWE: CWE-345
- Detect: In a Pipeline, a task producing IMAGE_URL/IMAGE_DIGEST results is from an untrusted source (git resolver with untrusted URL, or hub resolver). A compromised task can poison attestation.
- Parser: cross-reference result-producing tasks with trust status from TKN-TRUST checks
- Remediation: ensure all Chains-consumed result producers are from trusted, pinned sources

**TKN-CHAIN-007: Build pipeline without SBOM task**
- Severity: LOW
- CWE: CWE-1059
- Detect: Pipeline labeled as build type with no task named syft, cyclonedx, spdx, or sbom
- Risk: missing software bill of materials reduces supply chain transparency
- Remediation: add an SBOM generation task to the build pipeline

### Pipeline Logic (TKN-LOGIC-001..004)

New check module: `tekton_guard/checks/logic.py`

**TKN-LOGIC-001: Security task not in finally block**
- Severity: MEDIUM
- CWE: CWE-693
- Detect: Pipeline has a security-relevant task (scan, sign, verify, attest, clair, sast) in `spec.tasks` instead of `spec.finally`. If any preceding task fails, the security task is skipped.
- Risk: build failures silently skip security validation
- Remediation: move security tasks to `finally` block, or ensure they have no `runAfter` dependencies on failable tasks

**TKN-LOGIC-002: Overridable param default**
- Severity: MEDIUM
- CWE: CWE-1188
- Detect: Task defines a param with a security-relevant default (e.g., `--privileged=false`, `--skip-tls-verify=false`) that can be overridden by the PipelineRun caller
- Risk: PipelineRun can override secure defaults to insecure values
- Parser: check Task params with boolean defaults or flag-like defaults containing security keywords (privileged, tls-verify, skip, insecure, allow-all)
- Remediation: use hardcoded values instead of params for security-critical flags, or validate param values in the task script

**TKN-LOGIC-002 note**: this check has high FP potential. Security keywords in param names are heuristic. Recommend LOW severity and opt-in via config. Revised severity: LOW.

**TKN-LOGIC-003: TOCTOU via parallel workspace access**
- Severity: HIGH
- CWE: CWE-367
- Detect: In a Pipeline, two tasks share a workspace and can run in parallel (no `runAfter` dependency between them). If one is from an untrusted source, it can modify workspace data while the other reads it.
- Risk: untrusted task modifies source code or build artifacts between clone and build
- Parser: cross-reference workspace bindings with `runAfter` dependencies and trust status
- Remediation: add explicit `runAfter` ordering or use separate workspaces

**TKN-LOGIC-004: Pipeline without finally block**
- Severity: LOW
- CWE: CWE-390
- Detect: Pipeline with no `finally` block at all
- Risk: no cleanup, reporting, or error handling on pipeline failure
- Remediation: add a `finally` block with at minimum a status-reporting task

### Resolver Deep (TKN-TRUST-004..006)

Extends existing trust module.

**TKN-TRUST-004: HTTP resolver without digest**
- Severity: HIGH
- CWE: CWE-829
- Detect: Task or Pipeline referenced via `resolver: http` without a `digest` param. The HTTP resolver fetches YAML from arbitrary URLs without integrity verification.
- Risk: MITM or compromised web server can inject malicious task/pipeline definitions
- Remediation: add `digest: sha256:<hash>` param to HTTP resolver references

**TKN-TRUST-005: Cluster resolver in shared namespace**
- Severity: MEDIUM
- CWE: CWE-829
- Detect: Task referenced via `resolver: cluster` where the `namespace` param is a shared/multi-tenant namespace (configurable list, default: `tekton-pipelines`, `openshift-pipelines`)
- Risk: any user with Task create permission in the shared namespace can replace the referenced task
- Remediation: use a dedicated namespace for cluster-resolver tasks, or switch to bundle/git resolver with pinning

**TKN-TRUST-006: Bundle without VerificationPolicy**
- Severity: MEDIUM
- CWE: CWE-345
- Detect: Pipeline/PipelineRun uses `resolver: bundles` but no VerificationPolicy resource is found in the scanned files that covers the bundle's registry pattern
- Risk: bundle content is not signature-verified before execution
- Parser: cross-reference bundle references with VerificationPolicy resourcePattern rules
- Remediation: create a VerificationPolicy that requires signature verification for the bundle's registry

## Implementation

### Parser changes
- Add 4 new kinds to `TEKTON_KINDS`: TriggerTemplate, TriggerBinding, EventListener, Repository
- Add trigger-specific fields to `TektonResource` (or use `raw` dict for simpler checks)
- For checks that only need raw YAML access (CHAIN-005, TRUST-006), use `resource.raw` without new dataclass fields

### New check modules
- `tekton_guard/checks/logic.py`: TKN-LOGIC-001..004
- Extend `tekton_guard/checks/triggers.py`: TKN-TRIG-004..007
- Extend `tekton_guard/checks/chains.py`: TKN-CHAIN-003..007
- Extend `tekton_guard/checks/trust.py`: TKN-TRUST-004..006

### Config additions
```yaml
# Shared namespaces for cluster resolver trust check
shared_namespaces:
  - "tekton-pipelines"
  - "openshift-pipelines"

# Security-relevant task name patterns (for LOGIC-001, CHAIN-004)
security_task_patterns:
  - "scan"
  - "sign"
  - "verify"
  - "attest"
  - "cosign"
  - "enterprise-contract"
  - "sast"
  - "clair"
  - "clamav"
  - "sbom"
  - "syft"
  - "cyclonedx"
```

### Expected check count
- Existing: 28 checks (27 registered + 1 disabled TKN-LIMIT-001)
- New: 16 checks
- Total: 44 checks (43 registered)
- TKN-LOGIC-002 (overridable param default) registered but opt-in (disabled by default in skip_checks, same as TKN-LIMIT-001)

## Testing

Each new check gets at minimum 2 test fixtures:
- Positive fixture (finding triggered)
- Negative fixture (clean, no finding)

Complex checks (TRIG-004, CHAIN-006, LOGIC-003) get 3 fixtures (positive, negative, boundary).

Total: ~35 new test fixtures across the 16 checks.

## Phased Implementation

**Phase A**: Trigger Deep (TRIG-004..007) + parser extensions for TriggerTemplate/EventListener/Repository
**Phase B**: Supply Chain Deep (CHAIN-003..007) + VerificationPolicy scanning
**Phase C**: Pipeline Logic (LOGIC-001..004)
**Phase D**: Resolver Deep (TRUST-004..006)

Each phase is independently shippable. Phase A and B are highest value.
